# TDD Feature Skill v2

## Metadata

- **name**: tdd-feature
- **description**: Execute Test-Driven Development workflow following the 3-commit pattern (RED → GREEN → REFACTOR) with automatic file detection and scope inference
- **version**: 2.0.0

## Changelog from v1

- **Auto-detect existing patterns**: Analyzes similar files to match test structure
- **Smart scope inference**: Extracts scope from file path or feature description
- **Session logging**: Automatically logs execution to `.claude/skills/tdd-feature/SESSION_LOG.md`
- **Pre-validation**: Checks dependencies are installed before running tests
- **Format on save**: Runs Prettier before REFACTOR commit automatically

## Instructions

When the user invokes `/tdd-feature <description>`, follow this workflow:

### Phase 0: Auto-Detection

1. Parse the feature description to identify:
   - Target scope (auth, scan, care, bouquet, health, flowers, etc.)
   - File type (API route, component, utility)
   - Expected location based on existing patterns
2. Find similar existing files as templates:
   - API routes: `src/app/api/<scope>/*/route.ts`
   - Components: `src/components/*.test.tsx` patterns
   - Utilities: `src/lib/*.test.ts` patterns
3. Log start to SESSION_LOG.md

### Phase 1: RED Commit (Test First)

1. Create test file at appropriate location:
   - API routes: `src/app/api/<scope>/<action>/route.test.ts`
   - Components: `src/components/<Name>.test.tsx`
   - Utilities: `src/lib/<name>.test.ts`
2. Import the module being tested (will fail - expected)
3. Write tests describing expected behavior:
   - Follow existing describe/it patterns from similar files
   - Use proper mocking for external dependencies (Supabase, etc.)
   - Test error cases and success cases
4. Run `npm test` to confirm RED state (tests fail due to missing module)
5. Commit: `test(<scope>): [RED] <description>`

### Phase 2: GREEN Commit (Minimum Implementation)

1. Create implementation file to make tests pass
2. Follow patterns from similar existing files
3. Include proper error handling (500 for unexpected errors)
4. Run `npm test` to confirm GREEN state
5. Commit: `feat(<scope>): [GREEN] <description>`

### Phase 3: REFACTOR Commit (Clean Up)

1. Review code for improvements:
   - Remove duplication
   - Improve naming
   - Add JSDoc comments
   - Ensure type safety (no `any`)
2. Run `npm run format` for Prettier formatting
3. Run `npm test` to ensure tests still pass
4. Commit: `refactor(<scope>): <improvements made>`

### Phase 4: Final Verification

1. Run full test suite: `npm test`
2. Log completion to SESSION_LOG.md
3. Report summary to user

## Session Log Format

```markdown
## YYYY-MM-DD HH:MM - <feature-description>

- **Scope**: <scope>
- **Files**: `<test-file>`, `<impl-file>`
- **Commits**:
  1. `test(<scope>): [RED] <description>`
  2. `feat(<scope>): [GREEN] <description>`
  3. `refactor(<scope>): <improvements>`
- **Test Results**: X passed
- **Notes**: <any issues encountered>
```

## Constraints

- NEVER mix test and implementation in same commit
- ALWAYS follow 3-commit pattern
- Use `unknown` and narrow, never `any`
- Scope DB queries with `user_id = auth.uid()` for IDOR protection
- Run tests before each commit
- Prettier must pass before REFACTOR commit

## Example Usage

```
/tdd-feature "add logout endpoint"
/tdd-feature "create flower name validation"
/tdd-feature "implement rate limiting for scan"
```

## Expected Output

- 3 commits with proper messages
- Passing tests
- Formatted, refactored code
- Session log entry
